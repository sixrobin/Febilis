# Setup des transitions entre les tableaux.

-> Board Manager
	- s'occupe de faire les liens entre les triggers. Dans le tableau "Links", créer autant d'entrées qu'il y a de liens (une entrée = une paire de triggers).
	- les valeurs dans Transition View peuvent rester comme elles sont, comme l'indique le nom de la catégorie, c'est purement visuel, même si le "DownRespawnHeightOffset" change la position du joueur quand il respawn vers le bas, donc à prendre en compte vis-à-vis des colliders et tout ça.
	- Il faut paramétrer le InitBoard pour être le tableau dans lequel se trouve le joueur au lancement de la scène (j'ajouterai plus tard un truc pour détecter ça automatiquement).
	
-> Boards
	- chaque tableau est un gameObject ayant les caractéristiques suivantes :
		• le script Board attaché au parent, qui référence tous les BoardsLink enfants et les bounds de la caméra.	
			• (Il y a une méthode dans le ContextMenu pour récupérer les BoardsLink automatiquement en un clic)
		• les triggers des transitions (BoardsLink) en enfants.
		• un box collider qui représente les bounds de la caméra pour ce tableau.

		
-> Boards Link
	- script qui déclenche la transition vers un autre tableau dans OnTriggerEnter2D
	- à attacher donc à un Collider2D
	- paramétrer dessus la direction de SORTIE, la direction d'entrée sera automatiquement la direction opposée.
	- par défaut, le player est téléporté sur le transform du script. Mais, si le OverrideRespawnPos est référencé, alors on utilise ce transform.
		• /!\ Pour le respawn vers le haut, il est obligatoire de le référencer, sinon le joueur se téléporte sur le trigger, et tombe car il est dans le vide.
		
	Il y a deux prefabs prévus :
		• "Link Base" pour les transitions vers la gauche, la droite ou le haut.
		• "Link Exit Down" pour les transitions vers le bas, qui ajoute donc le transform de respawn pour le retour vers le haut.
		
		
		
		
Les transitions sont automatiques mais le déplacement du perso utilise le code normal, donc il faut faire attention aux collisions. Par exemple, un gros mur derrière un trigger va juste montrer le perso en train de courir sur place, ce qui sera moche même si la transition marchera.
Pareil pour les transitions vers le haut et le bas, il faut faire gaffe à ce que le joueur soit obligé de bien sauter ou bien tomber sans pouvoir se coincer ou gruger avec une roulade ou autre.

Il y a une commande VisualizeBoardsLinks true/false qui permet de visualiser les triggers in game.
